# 케이스 상태 워크플로우

`Case.상태`(접수 → 염색 → 검토 → 서명완료, [`ARCHITECTURE.md`](../../ARCHITECTURE.md) 데이터 모델)의 전이 규칙, 권한, 각 상태에서 가능한 작업을 정의합니다.
사용자 역할 정의는 [`PRODUCT_SENSE.md`](../../PRODUCT_SENSE.md) 참고 (판독의/병리사/관리자).

## 상태 정의

| 상태 | 의미 |
|---|---|
| **접수** | 케이스가 시스템에 등록됨. 환자/케이스/검체/블록 정보 입력 완료, 슬라이드 스캔은 아직 시작 전 |
| **염색** | 검체 염색 및 슬라이드 스캔이 진행 중. 슬라이드 업로드/메타데이터 추출이 이루어지는 단계 |
| **검토** | 케이스에 속한 슬라이드가 모두 준비되어 판독 대기/진행 중. 워크리스트에 노출됨 |
| **서명완료** | 판독의가 판독을 완료하고 서명함. 진단 결과가 확정됨 |

## 상태 전이 다이어그램

```
접수 ──→ 염색 ──→ 검토 ──→ 서명완료
                    ↑__________|
                    재오픈 (판독의 본인 또는 관리자)
```

- 정상 흐름은 접수 → 염색 → 검토 → 서명완료의 단방향 진행입니다.
- "서명완료"에서 "검토"로의 재오픈만 허용된 역방향 전이입니다 (그 외 역방향 전이는 없음).
- `Case.케이스 유형`(예: 동결절편 vs 일반 생검)에 따라 상태 전이 자체는 동일하며, `우선순위`로 처리 속도만 달라집니다. 동결절편도 급속 염색을 거치므로 "염색" 상태를 동일하게 통과합니다 — 케이스 유형별 별도 워크플로우는 두지 않습니다.

## 전이 규칙 및 권한

| 전이 | 트리거 | 가능한 역할 | 비고 |
|---|---|---|---|
| (생성) → 접수 | 케이스 등록 | 병리사, 관리자 | 환자/케이스/검체/블록 정보 입력 시점 |
| 접수 → 염색 | 첫 슬라이드 업로드 시작 | 병리사, 관리자 | 슬라이드 등록 시작과 함께 자동 전이 |
| 염색 → 검토 | "검토 요청" 액션 | 병리사, 관리자 | 케이스에 속한 모든 슬라이드의 업로드/메타데이터 추출이 완료된 경우에만 가능 |
| 검토 → 서명완료 | "서명" 액션 | 판독의, 관리자 | 담당 판독의만 가능 — [배정](#배정-assignment) 참고. `Case.진단 소견` 필수 입력 |
| 서명완료 → 검토 (재오픈) | "재오픈" 액션 | 판독의(서명한 본인), 관리자 | 재오픈 사유 입력 필수 |

## 각 상태에서 가능한 작업

| 상태 | 가능한 작업 |
|---|---|
| 접수 | 검체/블록/슬라이드 등록·수정, 환자정보 수정 |
| 염색 | 슬라이드 업로드 계속, 메타데이터 확인/수정 |
| 검토 | WSI 뷰어 열람, `Annotation` 작성/수정/삭제, `CellAnnotation`(AI) 조회, `Case.진단 소견` 작성/수정 |
| 서명완료 | WSI 뷰어 열람(읽기), `Annotation` 신규 작성·수정 불가 — 재오픈 전까지 |

## 배정 (Assignment)

- `Case`에 "담당 판독의" 필드를 둡니다 (배정 전에는 비어있음).
- **케이스 목록**: 모든 역할이 전체 케이스 목록을 조회할 수 있습니다. "나에게 배정된 케이스"는 필터로 좁혀볼 수 있습니다.
- **자가 배정(self-claim)**: 판독의가 "검토" 상태 케이스를 열어 판독을 시작하면(또는 명시적 "담당으로 지정" 액션) 본인이 담당 판독의로 설정됩니다.
- **재배정**: 판독의, 병리사, 관리자 모두 담당 판독의를 변경할 수 있습니다.
- **서명 권한**: "검토 → 서명완료"는 담당 판독의(+관리자)만 가능합니다.

## 워크리스트와의 관계

- 판독의 워크리스트에는 "검토" 상태의 케이스가 노출됩니다.
- 전체 케이스 목록 조회가 기본이며, "나에게 배정된 케이스" 필터로 좁혀볼 수 있습니다 ([배정](#배정-assignment) 참고).
- `Case.우선순위` 필드로 정렬됩니다.

## 감사 이벤트

모든 상태 전이는 `AuditLog`에 기록됩니다 ([`core-beliefs.md`](../design-docs/core-beliefs.md) 원칙 #2):

- 사용자, 타임스탬프, 이전 상태 → 이후 상태
- 재오픈의 경우 입력된 사유 포함
- 담당 판독의 배정/재배정 시에도 기록 (이전 담당자 → 신규 담당자)
- Case 상세 화면에는 최종 서명자/일시 및 재오픈 횟수를 요약 표시 (상세 이력은 AuditLog 조회로 확인)
