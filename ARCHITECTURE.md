# ARCHITECTURE.md

## 시스템 개요

디지털 병리 IMS는 모노레포 안의 3개 독립 서비스로 구성됩니다.

```
┌─────────────┐         ┌──────────────┐
│  frontend   │ ◄─────► │   backend    │──────┐
│  (Next.js)  │  REST   │ (Spring Boot)│      │
└──────┬──────┘  API    └──────┬───────┘      │
       │                        │              │
       │ 타일 요청               ▼              │
       │ (토큰 포함)     ┌──────────────────┐   │
       │                │  PostgreSQL +     │   │
       │                │  PostGIS          │   │
       │                └──────────────────┘   │
       ▼                                        ▼
┌─────────────┐         ┌─────────────────────┐
│ tile-server │ ──────► │ 스토리지 추상화 레이어  │
│ (Flask +    │         │ (WSI 원본 파일)        │
│  OpenSlide) │         │ 로컬 FS / S3 등        │
└─────────────┘         └─────────────────────┘
```

- **frontend**: 케이스/슬라이드 관리 UI, OpenSeadragon 기반 WSI 뷰어, 어노테이션 도구
- **backend**: 인증/인가, 케이스/슬라이드/어노테이션 도메인 로직, 감사로그, 뷰잉 토큰 발급, 파일 업로드
- **tile-server**: OpenSlide로 WSI 파일을 읽고, pyvips로 DZI(Deep Zoom Image) 타일을 생성/서빙. 토큰 검증만 수행하며 DB를 직접 조회하지 않음

## 핵심 아키텍처 결정

### 1. 배포 환경: 클라우드 + 스토리지 추상화 레이어

- 솔로 개발 환경의 운영 부담을 낮추기 위해 클라우드(국내 리전) 우선으로 진행하되, WSI 원본 파일 접근은 **추상화 레이어**를 통해 이루어집니다.
- 추상화 레이어는 "파일 키를 주면 로컬에서 읽을 수 있는 경로를 반환"하는 인터페이스이며, 구현체는 (a) 로컬/NAS 직접 마운트, (b) 오브젝트 스토리지(S3 등)에서 다운로드 후 로컬 캐시 — 두 가지를 지원합니다.
- 이를 통해 향후 온프레미스 전환 시에도 backend/tile-server 코드 변경 없이 구현체만 교체할 수 있습니다.
- 배경: 온프레미스는 환자 데이터 반출 규제와 네트워크 지연 면에서 유리하지만, 솔로 개발자에게 인프라 운영 부담이 큽니다. 추상화로 두 옵션을 모두 열어둡니다.

### 2. 프론트엔드 ↔ 타일서버 통신: 토큰 기반 직접 통신

WSI 뷰어는 줌/팬마다 수십~수백 건의 타일 요청을 발생시킵니다. 모든 요청이 backend를 거치면 병목이 되므로, **인증/감사는 backend가, 타일 전송은 tile-server가 직접** 담당합니다.

흐름:

1. frontend → backend: "슬라이드 #N 열람 요청"
2. backend: 권한 검증 → **감사로그 기록**(세션 시작) → 슬라이드 ID/사용자/만료시간을 담은 단기 서명 토큰(JWT) 발급 → DZI 디스크립터 + tile-server URL + 토큰 반환
3. frontend(OpenSeadragon) → tile-server: 토큰을 쿼리 파라미터로 포함해 타일 직접 요청
4. tile-server: JWT 서명/만료/슬라이드 ID 일치 여부를 로컬 검증(DB 조회 없음) → OpenSlide로 타일 서빙

추가 고려사항:

- **캐싱**: `Cache-Control: private` (브라우저 로컬 캐시만 허용, CDN/공유 캐시 금지)
- **토큰 갱신**: 만료 시 frontend가 backend에 자동 재발급 요청 (장시간 뷰잉 세션 지원) — 상세 설계는 추후 논의
- **감사 단위**: 타일 단위가 아닌 **세션(열람) 단위**로 기록 — 규제 요구사항(누가 언제 어떤 환자 데이터에 접근했는가)과 부합하면서 오버헤드도 적음

### 3. 데이터베이스: PostgreSQL + PostGIS (단일 RDB)

- NoSQL은 도입하지 않습니다. 솔로 개발 환경에서 DB 2종 운영의 트랜잭션 일관성/운영 비용이, 셀 단위 어노테이션(슬라이드당 ~10만 건) 규모에 비해 이득이 적습니다.
- PostGIS는 셀 어노테이션의 뷰포트 기반 공간 쿼리(GiST 인덱스, bounding box)를 위해 채택합니다.
- 향후 셀 어노테이션은 `ST_AsMVT()`로 Mapbox Vector Tile 형식 생성을 검토 — WSI 래스터 타일과 동일한 타일 좌표계 개념으로 오버레이 가능.

### 4. 타일 생성: OpenSlide(리더) + pyvips(타일 생성/인코딩)

- 이전 구성(OpenSlide 기반 DeepZoom 생성기, 기본 리사이즈/JPEG 품질 설정)으로 판독의 검증을 거친 결과, **세포 종류 구분에 타일이 선명하지 않다**는 피드백을 받음.
- 원인 후보: (a) 원본 파일에 없는 줌 레벨을 합성할 때의 저품질 리샘플링, (b) 타일 인코딩 JPEG 품질(Q) 값이 낮음.
- 결정: WSI 파일 디코딩은 계속 **OpenSlide**(다양한 스캐너 포맷 호환성)를 사용하되, 피라미드/타일 생성·인코딩은 **pyvips**(`dzsave`)로 전환.
  - pyvips는 OpenSlide를 로더로 활용 가능(`vipsload` → `openslideload`) — 디코딩 경로 자체는 유지되며 타일 생성·인코딩 단계만 교체.
  - 합성 레벨 리샘플링은 Lanczos 등 고품질 보간 사용.
  - 타일 인코딩 JPEG 품질(Q)을 상향 조정하고, 고배율/진단 영역은 PNG 무손실 출력도 검토.
  - 부수적으로 속도/메모리 효율도 기존 대비 개선 기대.
- 영향 범위: tile-server(Flask)의 타일 생성 파이프라인에 한정. backend와의 토큰 기반 통신 흐름(결정 #2)에는 영향 없음.

## 데이터 모델

```
Patient ─< Case ─< Specimen ─< Block ─< Slide ─< Annotation
                                          │
                                          └─< CellAnnotation (AI 생성, 별도 테이블)

User ──< AuditLog (모든 엔티티에 대한 접근/변경 기록)
```

| 엔티티 | 설명 / 주요 필드 |
|---|---|
| **Patient** | 환자번호(MRN), 이름, 생년월일, 성별 — MVP에서는 직접 입력/저장 (LIS 연동 없음) |
| **Case** | 접수번호, 환자 참조, 의뢰의, 케이스 유형, 상태(접수→염색→검토→서명완료), 우선순위 |
| **Specimen** | 케이스 참조, 검체 부위/명칭 |
| **Block** | 검체 참조, 블록 번호/라벨 |
| **Slide** | 블록 참조, 스토리지 키, 스캐너/포맷, 배율, **MPP(μm/px)**, 염색 종류, 스캔 상태, 썸네일/라벨 이미지 |
| **Annotation** | 판독의가 직접 작성. 슬라이드 참조, 작성자, 지오메트리(JSONB), 타입(영역/측정/메모), 측정값. 변경 이력은 AuditLog로 기록(별도 버전 테이블 없음) |
| **CellAnnotation** | AI 형태계측 결과. 슬라이드 참조, 좌표/지오메트리(point 또는 소형 polygon), 세포 분류, 형태계측 지표(JSONB). `slide_id` 기준 파티셔닝, 뷰포트 기반(bounding box) 쿼리로 조회 |
| **User** | 이름, 역할(판독의/병리사/관리자), 면허번호 |
| **AuditLog** | 사용자, 액션, 대상 리소스, 타임스탬프, IP — 세션 단위 열람 기록 포함 |

계층 구조는 `Case → Specimen → Block → Slide` 4단계로 모델링하되, 초기 UI에서는 단계적으로 노출합니다 (스키마 변경은 나중에 비용이 크므로 처음부터 전체 깊이를 반영).

## 데이터 흐름

### 슬라이드 업로드

1. frontend → backend: WSI 파일 업로드 (또는 업로드 URL 발급 후 직접 스토리지에 업로드)
2. backend: 스토리지 추상화 레이어를 통해 저장 → `Slide` 레코드 생성 (메타데이터 추출 — 배율/MPP/포맷 등)

### 슬라이드 열람

1. frontend → backend: 열람 요청 → 권한 검증 + 감사로그 + 토큰 발급
2. frontend → tile-server: 토큰으로 DZI 타일 직접 요청

### 어노테이션

1. frontend → backend: 어노테이션 생성/수정/삭제 요청 (슬라이드 픽셀 좌표 기준)
2. backend: 권한 검증 → DB 저장 → 감사로그 기록

### AI 셀 어노테이션 수집 (향후)

1. AI 파이프라인 → backend: 슬라이드별 셀 어노테이션 결과를 `COPY`로 일괄 적재
2. frontend → backend: 뷰포트 bounding box로 `CellAnnotation` 조회 (PostGIS 공간 쿼리)

## 미결정 / 추후 논의 사항

- 인증 방식 전반 (세션 vs JWT, SSO 연동 여부)
- 토큰 갱신 흐름 상세 설계
- 스토리지 추상화 레이어의 구체적 인터페이스/구현 라이브러리
- 배포 인프라 상세 (컨테이너 오케스트레이션, CI/CD)
- pyvips 타일 인코딩 세부 파라미터 (JPEG Q값, PNG 적용 범위/배율 기준, 타일 크기/오버랩)
