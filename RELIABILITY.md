# RELIABILITY.md

## 신뢰성의 정의: 데이터 내구성 > 가용성

이 시스템에서 "신뢰성"의 우선순위는 **가용성(uptime)보다 데이터 내구성(durability)**입니다. 몇 시간의 다운타임은 복구 가능하지만, 환자 진단 데이터(`Case`, `Specimen`, `Block`, `Slide`, `Annotation`, `AuditLog`, WSI 원본 파일)의 손실은 복구할 수 없습니다 ([`core-beliefs.md`](docs/design-docs/core-beliefs.md) 원칙 #1).

솔로 개발 환경의 운영 부담을 고려해, 24/7 고가용성 인프라(다중 인스턴스, 자동 failover)는 MVP 범위에 두지 않습니다 ([`core-beliefs.md`](docs/design-docs/core-beliefs.md) 원칙 #4). 대신 데이터를 잃지 않는 백업/복구 체계를 우선합니다.

## 가용성 목표

- 단일 인스턴스/단일 리전으로 시작합니다.
- 업무시간(평일) 중 장애는 가능한 빠르게 대응하되, 야간/주말 장애는 다음 영업일 복구를 허용합니다.
- 사용 규모(동시 사용자 수, 기관 수)가 커지면 이 목표를 재검토합니다.

## 데이터 내구성 / 백업

### PostgreSQL

- 일일 스냅샷 + PITR(Point-in-Time Recovery)을 설정합니다 (예: 7일 보존 — 정확한 기간은 [`audit-log.md`](docs/product-specs/audit-log.md) 보존정책과 함께 재검토).
- 백업은 운영 DB와 분리된 위치(별도 리전/스토리지)에 보관합니다.

### WSI 원본 파일

- 스토리지 추상화 레이어([`ARCHITECTURE.md`](ARCHITECTURE.md) 결정 #1)의 구현체로 선택하는 클라우드 오브젝트 스토리지의 내장 내구성에 의존합니다 (별도 백업 불필요).
- 실수 삭제 방지를 위해 버전닝 또는 일정 기간의 soft-delete(휴지통)를 적용합니다.
- 업로드 시 체크섬을 기록해 전송 중 손상을 감지합니다.

### 복구 절차 검증

- 백업으로부터의 복구는 최소 연 1회 실제로 수행해 검증합니다. 백업이 "있다"는 것과 "복구된다"는 것은 다릅니다.

## 장애 등급 (Severity)

| 등급 | 정의 | 예시 | 대응 |
|---|---|---|---|
| Critical | 진단 업무 자체가 불가능 | DB 다운, 로그인 불가, 슬라이드 열람 전체 실패 | 즉시 대응, 영업시간 내 복구 목표 |
| Major | 핵심 기능 일부 장애 | 특정 슬라이드 타일 생성 실패, 어노테이션 저장 실패, 서명 불가 | 영업일 내 복구 |
| Minor | 부가 기능/표시 문제 | UI 스타일 오류, 통계 화면 오류 | 다음 배포에서 수정 |

## 모니터링 / 알림

- 별도 APM 도구 도입보다, 클라우드 제공자가 기본 제공하는 모니터링/헬스체크를 우선 활용합니다 ([`core-beliefs.md`](docs/design-docs/core-beliefs.md) 원칙 #4).
- 핵심 헬스체크 대상: backend API, tile-server, DB 연결.
- Critical 등급 장애는 솔로 개발자에게 즉시 알림이 가야 합니다 — 구체적 채널은 [미해결](#미해결--추후-논의).

## 타일 캐시 신뢰성

- 업로드된 WSI 원본 파일은 불변(immutable)입니다. DZI 타일은 tile-server가 요청 시점에 생성해 캐시하는 산출물로, 원본으로부터 언제든 재생성 가능합니다 ([`ARCHITECTURE.md`](ARCHITECTURE.md) 결정 #4).
- 타일 생성 실패는 조회 시점에 발생합니다 — 뷰어는 해당 타일에 오류 표시와 재시도 옵션을 제공합니다.
- 메타데이터/썸네일/ICC 프로필 추출 실패는 업로드 단계에서 감지되며, `Slide`의 처리 상태를 "실패"로 표시합니다 — 이는 [`case-status-workflow.md`](docs/product-specs/case-status-workflow.md)의 "염색 → 검토" 전이 조건과 연결됩니다.
- 타일 인코딩 파라미터(JPEG Q값 등)를 변경하면 캐시를 무효화하고 재생성합니다 — 원본이 보존되어 있으므로 안전합니다.
- 캐시는 조회된 영역만큼만 누적되어 사용량 기반으로 증가합니다. 장기적으로 캐시 용량이 커지면 LRU 등 정리(eviction) 정책을 고려할 수 있습니다 — [미해결](#미해결--추후-논의).

## 배포 / 마이그레이션 안전성

- DB 스키마 마이그레이션은 가능한 backward-compatible 방식(컬럼 추가 우선, 즉시 삭제 지양)을 사용해 무중단 배포를 지향합니다.
- PHI/PII 관련 스키마 변경은 [`AGENTS.md`](AGENTS.md) 가드레일에 따라 사용자 확인을 거칩니다.
- 배포 실패 시 이전 버전으로 재배포하는 롤백 절차를 둡니다.

## 미해결 / 추후 논의

- Critical 장애 알림 채널 (이메일/SMS 등)
- DB 백업 보존 기간의 정확한 값 — `audit-log.md` 보존정책과 함께 결정
- 재해 복구(DR, 리전 장애 대응) — MVP 범위 밖, 사용 규모 확대 시 재논의
- 구체적 RTO/RPO 수치
- 타일 캐시 정리(eviction) 정책 — 사용량 기반으로 증가, 필요 시 LRU 등 검토
